# 검토 전 자가검증 게이트 (Pre-Review Verification Gate)

| 항목 | 내용 |
|---|---|
| **목적** | Claude(Chat/Cowork/Code 전 탭)가 반복해서 틀린 패턴을 **결론·문서·고객멘트 산출 직전**에 강제 점검한다 |
| **적용** | 모든 결론/분석/문서/멘트 출력 직전. 게이트 10문항 중 **No가 하나라도 있으면 출력 보류** → 해결 또는 `[확인 필요]` 명시 후 진행 |
| **근거** | 2026-06 이전 실제 대화에서 반복 검출된 오류 패턴 |
| **최종수정** | 2026-06-30 |

> 이 문서는 engr-jira 소유 전 리포에 동일 사본으로 박혀 있다(공통 표준). 코드/배포와 무관한 작업 지침 문서.

## 1. 반복 이슈 리스트 (R-01 ~ R-10)

| ID | 반복 패턴 | 실제 사례 | 무엇이 틀렸나 |
|---|---|---|---|
| **R-01** | 분석 대상 식별자(IP·호스트·포트·계정·캡처지점) 미확정 상태에서 결론 단정 | 와이어샤크 SMTP: 9080 흐름(120↔121)을 SEPM으로 단정, 실제 SEPM=172.16.1.205. "구조상 확정"이라 박았다가 IP 듣고 번복 | 가장 먼저 물었어야 할 대상 IP를 안 묻고 추정으로 전체 분석을 쌓음 |
| **R-02** | 현재 운영 baseline 미확보 → 하향(regression) 제안 | 우리카드 HW 스펙: 현재 DB 128GB인데 64GB 제안 | 재구축인데 현 운영 실측을 baseline으로 안 받고 일반 스케일다운 |
| **R-03** | 가설을 "확정/구조상 확정"으로 표기 (사실↔가설 경계 붕괴) | 와이어샤크 "추정 아님, 구조상 확정" 외 다수 | 추론을 관측처럼 단정. 답변원칙(사실/[가설] 구분) 위반 |
| **R-04** | 고객 전달용에서 "불가능/범위 외" 단정 | 처브라이프 CRX "개발이라 범위 외" / DLP 확장 "제품 한계" → 빠꾸 | "가능+선행조건, 제약은 외부요인" 원칙 반복 위반 |
| **R-05** | 환경 출처 혼동 (랩 값 ↔ 고객 값) | ENGR-3225: scm.server.name=B205WIN2022(랩)를 고객(SAM-SEC-SEP11) 분석에 혼입 | 인용 설정값의 환경 라벨 없이 결론에 사용 |
| **R-06** | 결정적 필수 데이터 누락 상태로 결론 진행 | ENGR-3261: SEP 클라이언트 빌드 미수집인데 가설 확정 톤 | "없으면 확정 불가"를 선표시 안 함 |
| **R-07** | 사용자가 아는 범위를 넘어선 단정 멘트 생성 | 처브라이프: "세부 모른다"는데 "예외처리 됐던 거 확인했다"식 멘트 | 검증 안 된 것을 사용자 입에 단정으로 넣음 |
| **R-08** | 벤더/KB 단정 무검증 수용 | (반례·기준선) ENGR-3225: CRE-18728 벤더 단정을 랩 재현으로 검증·반박 | 우리 환경 재현·로그 근거 없이 벤더 단정 수용 금지 |
| **R-09** | 호칭·고정사실 오류 | 채기태=팀장님(부장 아님) 반복 정정 | 고정 사실을 메모리 기준과 미대조 |
| **R-10** | 전수검사·이상유무 판정을 결과물 표시값만으로 수행 | 처브라이프 ③: "차단정책 다 봤다"지만 실제 Discover 일부만 봄 → "이상 없음" 오판 | 결과물/콘솔/요약 표시 범위만 보고 전수 판단. 원본 전량과 미대조 |

## 2. 검토 전 게이트 — 10문항 (출력 직전 강제 통과)

> 규칙: 각 항목 **Yes**여야 출력. **No가 하나라도** 있으면 출력 보류 → 사용자 확인 요청 또는 `[확인 필요]` 명시 후 진행.

1. **대상 확정 (R-01)** — 분석에 쓴 IP/호스트/포트/계정/환경을 사용자가 확정했는가? 추정 식별자엔 `[확인 필요]`를 달았는가?
2. **baseline 확보 (R-02)** — (설계·스펙류) 현 운영 실측을 받았는가? 제안값이 현재보다 낮은 항목이 있으면 "의도된 하향"임을 명시했는가?
3. **사실/가설 분리 (R-03)** — 각 문장이 직접 관측(로그·캡처·공식KB)인가 추론인가? 추론엔 `[가설]`을 붙였는가? 근거 없는 "확정/무조건"을 쓰지 않았는가?
4. **고객 프레임 (R-04)** — "불가능/안 됨/범위 외" 단정이 있는가? → "가능+선행조건"으로 전환, 제약은 외부요인(브라우저정책·표준) 귀속, "다른 기능 정상" 한정 문구 포함했는가?
5. **환경 라벨 (R-05)** — 인용 값·스크린샷이 랩(정상)인지 고객(장애)인지 라벨을 달았는가? 두 환경 값을 섞지 않았는가?
6. **누락 선표시 (R-06)** — 진단에 결정적 데이터가 빠졌으면, 결론 위에 "이 항목 없으면 확정 불가"를 먼저 박았는가?
7. **사용자 인지 범위 (R-07)** — 사용자가 모른다고 한 항목을 "확인했다"로 적지 않았는가?
8. **벤더 검증 (R-08)** — 벤더/KB 단정을 우리 환경에서 재현·로그로 확인했는가? "왜 우리 환경이 그 결함인지" 근거가 있는가?
9. **고정사실 대조 (R-09)** — 호칭(기태 팀장님)·팀원명·고객사·리포 규칙(claude-cowork / engr-hub dev→prod)을 메모리와 대조했는가?
10. **전수=실데이터 매칭 (R-10)** — 전수/이상유무 판정을 결과물 표시값만으로 했는가, 아니면 연결된 원본 실데이터(DB 레코드 전량·전체 incident·원본 로그 전수)와 대조했는가? 표시 건수와 원본 건수가 일치하는가? 표본만 봤으면 "전수 아님"을 명시했는가?

---

## 3. 운영 규칙

- 신규 반복 패턴 발견 시 **R-11 이후**로 추가하고 게이트 문항도 갱신한다.
- R-10은 특히 전수검사(매체제어 정책·incident·로그 점검)에서 "표시값=실제" 가정을 금지하는 항목이다. 항상 **연결된 원본 데이터의 전체 건수와 대조**한다.
- 코드/배포 변경 워크플로우(요청→제안→승인→dev→검증→명시 요청 시 prod)는 이 게이트와 별도로 그대로 유지된다.
